PRD - WhatsApp Engineer

Skill OpenClaw para o OpenClaw agent resumir conversas do WhatsApp e entregar o resultado no destino escolhido pelo usuario.

1. Objetivo

Criar uma skill operacional para o agente OpenClaw ler historico local do WhatsApp via MCP whatsapp-profiles, gerar resumos de texto, links, arquivos e audios, e entregar esses resumos no local definido pelo usuario.

O produto deve funcionar como camada de orquestracao: a skill nao substitui o MCP de WhatsApp, nem os conectores de entrega. A transcricao de audio e nativa na skill via scripts/transcribe_audio.py. O PRD define o fluxo, o formato de configuracao, os guardrails e o comportamento esperado.

2. Usuarios E Casos De Uso

Usuario Necessidade Resultado esperado
Dono/operador do OpenClaw agent Receber resumo diario ou sob demanda de grupos e chats importantes. Mensagem consolidada em Telegram, WhatsApp, Discord ou destino manual.
Equipe comercial Acompanhar pendencias, clientes, orcamentos, audios e links enviados por vendedores. Resumo com decisoes, responsaveis e proximas acoes.
Equipe administrativa Monitorar documentos, pagamentos, demandas e riscos em grupos de WhatsApp. Resumo operacional com anexos e itens acionaveis.

3. Escopo

Incluido

  • Wizard para configurar origem, periodo, audio, formato e destino de entrega.
  • Leitura de mensagens via MCP whatsapp-profiles.
  • Inventario de links, imagens, videos, documentos, PDFs e audios.
  • Download de midia quando a bridge do perfil estiver aberta.
  • Transcricao de audio nativa via scripts/transcribe_audio.py e faster-whisper.
  • Entrega configuravel em Telegram, WhatsApp, Discord, manual ou custom.
  • Guardrails para nao salvar tokens, QR Codes, sessoes, bancos SQLite ou midias privadas na skill.

Fora Do Escopo Atual

  • Enviar mensagens por WhatsApp sem adaptador/tool explicito de envio.
  • Rodar agendamento persistente sem um scheduler externo do OpenClaw, cron, systemd ou TaskFlow.
  • Hospedar webhook publico para Telegram ou Discord dentro da skill.
  • Armazenar historico completo de resumos dentro do repositorio da skill.

4. Arquitetura

  1. Usuario executa o wizard scripts/configure.py.
  2. Wizard grava ~/.openclaw/state/whatsapp-engineer/config.json com permissao 0600.
  3. OpenClaw agent consulta perfis e mensagens via MCP whatsapp-profiles.
  4. OpenClaw agent lista assets e baixa audios relevantes quando necessario.
  5. OpenClaw agent transcreve audio via scripts/transcribe_audio.py.
  6. OpenClaw agent monta o resumo conforme references/digest-format.md.
  7. OpenClaw agent entrega no destino definido em destinations.primary.
  8. Para Telegram, scripts/test_delivery.py --send pode medir o tempo da chamada ao Bot API.
WhatsApp local DB
  -> whatsapp-profiles MCP
  -> OpenClaw agent / whatsapp-engineer
  -> audio transcription, digest, safety checks
  -> Telegram | WhatsApp adapter | Discord | manual/custom

5. Wizard

O wizard deve coletar configuracao operacional sem pedir segredo direto.

Campo Descricao Exemplo
source.profiles Perfis do WhatsApp a consultar. Vazio significa todos. vendedor-joao, financeiro
source.chats Chats, grupos ou JIDs especificos. Vazio significa todos filtrados pelo periodo.
source.default_period Periodo padrao de busca. last_24h
audio.transcribe Define se audios relevantes serao baixados e transcritos. true
destinations.primary Destino escolhido pelo usuario. telegram, whatsapp, discord

6. Destinos De Entrega

Telegram

  • Suporta grupo, canal, chat privado e topico/forum.
  • Usa chat_id e, para topico, message_thread_id.
  • Token deve ficar em variavel de ambiente, por exemplo TELEGRAM_BOT_TOKEN.
  • Teste de entrega mede latencia da chamada HTTP, nao confirmacao de leitura.

WhatsApp

  • Suporta configuracao para grupo, direto, canal ou lista.
  • Exige profile_slug, chat_jid e um send_adapter.
  • Limitacao atual: o MCP whatsapp-profiles do kit e leitura/download; envio real exige adaptador separado.

Discord

  • Suporta webhook ou bot adapter.
  • Webhook URL ou bot token devem ficar em variaveis de ambiente.
  • Mensagens devem usar Markdown simples e respeitar limite da plataforma.

Manual/Custom

  • Gera o resumo e marca publicacao como pendente ou delegada.
  • Serve para Slack, email, Notion, CRM ou outro destino futuro.

6.1 Multiplos Numeros E Funcionarios

O WhatsApp Engineer deve suportar analise de varios numeros usando o modo perfis do whatsapp-mcp-local-kit. Cada funcionario deve ser cadastrado como um perfil separado, com sessao, porta local, whatsapp.db e messages.db proprios. 

1 numero de WhatsApp = 1 perfil
1 funcionario = normalmente 1 perfil
varios funcionarios = varios perfis em source.profiles

Fluxo De Cadastro

  1. Abrir o painel WhatsApp MCP Tray.
  2. Criar um projeto, como Vendedores, Financeiro ou Suporte.
  3. Criar um perfil por funcionario, como vendedor-joao ou financeiro-ana.
  4. Clicar em Conectar QR.
  5. O funcionario escaneia o QR em WhatsApp > Aparelhos conectados > Conectar aparelho.
  6. Aguardar a primeira sincronizacao inteligente.
  7. Repetir para cada funcionario.

Analise Pelo OpenClaw agent

O OpenClaw agent deve consultar list_profiles, filtrar os perfis definidos em source.profiles e agrupar os resultados por projeto, perfil/funcionario, grupo/chat, remetente e periodo. Se source.profiles estiver vazio, o OpenClaw agent pode analisar todos os perfis habilitados.

Guardrails

  • Funcionario precisa autorizar o pareamento por QR.
  • Nao misturar sessoes, bancos ou pastas entre funcionarios.
  • Nao apagar bancos ou sessoes sem confirmacao explicita.
  • Indicar no resumo qual perfil/funcionario originou cada item importante.

8. Requisitos Nao Funcionais

  • Privacidade: nao salvar tokens, QR Codes, bancos SQLite, sessoes, logs reais ou midias privadas na pasta da skill.
  • Seguranca: usar variaveis de ambiente para tokens e webhooks.
  • Auditabilidade: config JSON legivel e versionado por schema, mas fora do Git.
  • Portabilidade: manter adaptadores por plataforma separados.
  • Resiliencia: quando uma midia nao puder ser baixada, resumir o contexto textual e apontar o bloqueio.
  • Clareza: todo resumo deve indicar origem e periodo.

10. Formato Do Resumo

  1. Cabeçalho com periodo, origem e destino.
  2. Principais pontos consolidados.
  3. Pendencias com responsavel e prazo quando identificado.
  4. Decisoes tomadas ou indicios de decisao.
  5. Audios resumidos com horario e remetente.
  6. Links e arquivos acionaveis.
  7. Riscos, bloqueios e alertas.

11. Guardrails

  • Confirmar a primeira entrega em qualquer destino de grupo/canal.
  • Exigir confirmacao explicita para envio em WhatsApp.
  • Mascarar telefones quando safety.redact_phone_numbers estiver ativo.
  • Se o volume de mensagens for alto, pedir recorte por periodo, perfil ou chat antes de publicar.
  • Separar fatos de inferencias no resumo.
  • Nao prometer envio por WhatsApp sem adaptador de envio instalado e testado.

13. Roadmap

  1. Adicionar executor de digest sob demanda usando o config gerado.
  2. Adicionar scheduler via TaskFlow, cron ou systemd user timer.
  3. Implementar adaptador de envio por WhatsApp quando a tool de envio estiver disponivel.
  4. Adicionar suporte a multiplos destinos simultaneos.
  5. Adicionar historico local opcional de resumos, sem mensagens brutas.
